# Multi-Year Consistency Cleanup Report

## Scope

This cleanup pass targeted the highest-value Multi-Year consistency issues identified during the documentation pass:

1. Investment-step key convention mismatch
2. Year-label policy mismatch
3. Battery degradation schema ownership ambiguity
4. Legacy Multi-Year schema branches still accepted by loaders

## What Changed

### 1. Aligned Multi-Year step keys with internal investment-step labels

- Updated Multi-Year template generation so user-facing YAML step keys now use canonical numeric labels:
  - single-step: `"1"`
  - multi-step: `"1"`, `"2"`, ...
- Kept centralized low-cost backward compatibility in the loader for:
  - `step_1`, `step_2`, ...
  - single-step `base`
- Tightened remapping behavior so `base` is no longer silently broadcast in multi-step projects.

Why:

- The internal Multi-Year sets already use numeric `inv_step` labels.
- Using the same convention in templates removes unnecessary hidden remapping.
- Rejecting multi-step `base` avoids surprising defaults.

### 2. Made Multi-Year year handling explicitly integer-based

- Updated the Streamlit Project Setup page to collect the Multi-Year start year via a numeric input instead of a free-text label field.
- Updated template year generation to require an integer-like start year and fail fast otherwise.
- Improved the set-initialization validation message for invalid `start_year_label`.

Why:

- The current Multi-Year implementation fundamentally uses integer-like calendar years.
- The previous template helper implied that non-integer fallback labels might work, which was misleading.
- Failing early in the UI is clearer than failing later during dataset initialization.

### 3. Clarified battery degradation schema ownership

- Stopped writing battery-specific degradation detail fields into `formulation.json`.
- Kept only the high-level activation flags there:
  - `cycle_fade_enabled`
  - `calendar_fade_enabled`
- Updated `battery.yaml` template generation to include `battery.technical.initial_soh` when endogenous degradation is enabled.
- Required `battery.technical.initial_soh` in the Multi-Year battery loader when endogenous degradation is active.
- Removed the formulation-level requirement that calendar-fade curve filenames must already be present in `formulation.json`; the current workflow now expects that battery-specific path in `battery.yaml`.

Why:

- Battery degradation parameters belong with the battery technology input, not with the project-structure file.
- This makes the ownership model clearer:
  - `formulation.json` activates features
  - `battery.yaml` stores the battery’s actual degradation inputs

### 4. Removed stale legacy Multi-Year schema branches

- Multi-Year renewables loader now rejects:
  - top-level `by_step`
  - `technical.by_step`
  - `operation`
- Multi-Year battery loader now rejects:
  - top-level `by_step`
  - `technical.by_step`
  - `operation`
- Multi-Year generator loader now rejects:
  - top-level `by_step`
  - `technical.by_step`
  - `operation`
- Multi-Year fuel loader now rejects:
  - `fuel.by_step`
  - legacy `fuel.by_scenario`

Why:

- These branches are no longer generated by the UI.
- Keeping them silently accepted increased ambiguity and code weight.
- Explicit errors are cleaner and easier to document than hidden fallback behavior.

## Behavior Before vs After

### Investment-step keys

- Before:
  - templates used `base` or `step_1`, `step_2`, ...
  - sets used `1`, `2`, ...
  - parser silently remapped between them
- After:
  - Multi-Year templates use `1`, `2`, ...
  - loader still accepts legacy aliases centrally for backward compatibility
  - multi-step use of `base` now raises a clear error

### Year labels

- Before:
  - UI accepted a free-text start-year label
  - template helper implied fallback labels like `year_1` might work
  - set initialization later required integer-like years
- After:
  - UI enforces numeric start years directly
  - template generation also requires integer-like year labels
  - validation messages clearly state the calendar-year requirement

### Battery degradation ownership

- Before:
  - degradation detail was split awkwardly between `formulation.json` and `battery.yaml`
  - `initial_soh` was not part of the generated battery template
- After:
  - `formulation.json` carries only activation flags
  - `battery.yaml` carries battery-specific degradation parameters
  - `initial_soh` is part of the generated degradation-aware battery schema

### Legacy Multi-Year schema

- Before:
  - loaders still accepted several outdated schema variants
- After:
  - those outdated branches are rejected with explicit validation messages pointing users to the current shared-technology schema

## Backward Compatibility Intentionally Preserved

- Multi-Year step remapping still accepts:
  - `step_1`, `step_2`, ...
  - single-step `base`
- Battery calendar-fade input still accepts the legacy `calendar_fade_coefficient_per_step` CSV column alias.
- Battery calendar time increment still accepts legacy `per_step` naming aliases in the parser.
- Generator and battery curve utilities still accept legacy absolute-efficiency curve interpretations where already supported by shared utilities.

## Remaining Open Items

1. Calendar-fade time increment still carries legacy `per_step` aliases.
2. Exogenous battery degradation is only suppressed automatically when calendar fade is enabled.
3. `grid_availability.csv` is still stored inside `inputs/` even though it is derived.
4. Vintage-label plumbing exists but is not yet surfaced as a clean primary workflow.
5. Result-export naming could still benefit from a more formal external reporting convention.
