# Typical Year Consistency Cleanup Report

## Scope

This cleanup pass targeted the highest-value Typical Year consistency issues identified during the documentation pass:

1. Battery `initial_soh` template/parser mismatch
2. Typical Year generator parser requiring `investment.by_step.base` more strictly than other loaders
3. Generator curve zero-anchor wording mismatch
4. Legacy Typical Year schema branches still accepted by parsers but no longer generated by the UI

## What Changed

### 1. Removed dead Typical Year `initial_soh` path

- Removed `battery_initial_soh` from the Project Setup configuration flow.
- Stopped writing `initial_soh` into `formulation.json` from the Streamlit page.
- Removed `battery_initial_soh` from Typical Year parameter plumbing.
- Updated the Typical Year battery parser to reject `battery.technical.initial_soh` explicitly.

Why:

- Typical Year does not use battery SoH state.
- The UI did not expose a real user control for this value.
- Keeping it in Typical Year paths created a misleading parser/template mismatch.

### 2. Harmonized Typical Year single-step parsing

- Added a shared Typical Year helper that accepts:
  - `investment.by_step.base`, or
  - exactly one non-`base` step block
- Applied that logic to Typical Year renewable, battery, and generator loaders.

Why:

- Typical Year only needs one investment block.
- Requiring `base` only for generators was stricter than necessary and inconsistent with the rest of the steady-state workflow.
- Silently taking an arbitrary first step was also too ambiguous.

### 3. Clarified generator zero-anchor behavior

- Kept the robust parser behavior that accepts either:
  - an explicit zero-output row, or
  - positive-load points only
- Updated wording in generated input help and user docs to match the actual parser behavior.

Why:

- This is the most user-friendly option.
- The helper template already includes a `0.0` row, but the parser also supports positive-only input safely.

### 4. Removed legacy Typical Year operation blocks

- Typical Year parsers now reject legacy `operation` blocks in:
  - `renewables.yaml`
  - `battery.yaml`
  - `generator.yaml`
- Current fixed O&M, subsidy, and shared curve settings must now be provided only through the current generated schema.

Why:

- These branches were no longer generated by the UI.
- Keeping them silently accepted added ambiguity and code clutter.
- Explicit errors are cleaner than hidden fallback behavior.

## Behavior Before vs After

### Battery `initial_soh`

- Before:
  - Typical Year templates did not generate `battery.technical.initial_soh`
  - Typical Year parser still accepted it and defaulted it to `1.0`
  - Streamlit setup carried a dead `battery_initial_soh` path
- After:
  - Typical Year no longer exposes or accepts `initial_soh`
  - The parser raises a clear validation error if it is present

### Typical Year investment-step handling

- Before:
  - renewables and battery accepted the first available step
  - generator required `base`
- After:
  - all Typical Year loaders accept `base` or exactly one step block
  - multiple non-`base` steps now raise a clear error

### Generator curve zero row

- Before:
  - parser behavior and wording were not fully aligned
- After:
  - parser behavior is unchanged but explicitly documented:
    - one zero anchor is always used internally
    - user may include or omit the explicit zero row

### Legacy Typical Year schema

- Before:
  - parsers still accepted old `operation.by_scenario` branches
- After:
  - those branches are rejected with explicit messages directing users to the current schema

## Backward Compatibility Intentionally Preserved

- Generator and battery curve files still accept the legacy absolute-efficiency interpretation where that behavior is already handled by shared curve utilities.
- Generator curve input still accepts both explicit-zero and positive-load-only support points.

## Remaining Open Items

1. Typical Year still shares some high-level Project Setup and `formulation.json` fields with the dynamic workflow.
2. Land-limit usability still depends on users filling renewable area coefficients consistently.
3. `grid_availability.csv` remains a derived artifact that users can still edit manually outside the app.
